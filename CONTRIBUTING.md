# Contributing to PUNN Cognitive Architecture (PCA)

Thank you for your interest in contributing to the PUNN Cognitive Architecture (PCA).

PCA is an open, implementation-independent cognitive architecture dedicated to advancing trustworthy, constitution-centered intelligence that strengthens Human Agency.

We welcome contributions from researchers, developers, educators, designers, students, and organizations.

---

# Guiding Principles

All contributions should preserve the architectural principles of PCA.

In particular, contributors are expected to respect the following priorities:

- Constitution Before Optimization
- Human Agency Above Automation
- Understanding Before Generation
- Evidence Before Confidence
- Transparency Before Persuasion
- Deliberation Before Commitment
- Continuous Learning
- Responsible Creativity
- Constructive Criticism
- Responsible Evolution

These principles take precedence over implementation preferences.

---

# Ways to Contribute

You can contribute by:

- Improving documentation
- Correcting errors
- Proposing architectural improvements
- Implementing reference components
- Writing examples and tutorials
- Developing benchmarks
- Improving developer tools
- Reporting bugs
- Reviewing pull requests
- Conducting research based on PCA

Every contribution is valuable when it improves understanding and architectural quality.

---

# Before Opening an Issue

Please check whether:

- the issue has already been reported;
- it is related to the PCA Specification or Reference Implementation;
- sufficient information is provided.

When reporting an issue, include whenever possible:

- Specification version
- Component or chapter
- Environment (if applicable)
- Expected behavior
- Actual behavior
- Steps to reproduce
- Suggested improvement

---

# Pull Requests

Pull requests should:

- Preserve constitutional principles.
- Maintain architectural consistency.
- Include clear documentation.
- Explain the motivation for significant changes.
- Avoid unnecessary complexity.
- Follow the existing repository structure.

Large architectural changes should be discussed through an Issue before implementation.

---

# Documentation

Documentation should:

- Be written in clear English.
- Use terminology defined in the PCA Specification.
- Remain implementation-independent whenever practical.
- Explain architectural decisions rather than only implementation details.

---

# Coding Guidelines

Reference implementations should emphasize:

- Readability
- Simplicity
- Modularity
- Explainability
- Maintainability
- Testability

Architecture should guide implementation, not the reverse.

---

# Architectural Proposals

Major architectural proposals should include:

- Problem statement
- Motivation
- Proposed solution
- Constitutional impact
- Compatibility considerations
- Expected benefits
- Possible risks

Architectural changes should be evidence-based whenever possible.

---

# Testing

Contributors are encouraged to include tests for new functionality.

Tests may include:

- Unit tests
- Integration tests
- Behavioral tests
- Compliance tests
- Benchmark evaluations

Testing should validate architectural behavior as well as software correctness.

---

# Code of Conduct

Please communicate respectfully and professionally.

Discussion should focus on ideas, evidence, and architectural quality rather than individuals.
